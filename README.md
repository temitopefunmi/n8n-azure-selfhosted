# Self-hosting n8n on Azure (Terraform, Docker, NGINX, Key Vault)

This repository deploys a **production-ready, self-hosted n8n instance on Azure** using **Terraform**, **Docker Compose**, **NGINX**, and **Azure Key Vault**.

The goal is simple:

> Run `terraform apply` and immediately access n8n over HTTPS — with secrets generated securely, stored in Key Vault, and fetched at runtime using Managed Identity.

No SSH required. No manual secret handling. Fully automated boot.

---

# Architecture Overview

**Infrastructure:**

* Azure Linux VM (Ubuntu)
* System-assigned Managed Identity
* Network Security Group (22, 80, 443)
* Azure Key Vault

**Runtime Stack (on VM):**

* Docker
* Docker Compose
* n8n container
* PostgreSQL container
* NGINX (host-level reverse proxy with TLS)

```
Internet
   ↓
NGINX (VM, port 443)
   ↓
localhost:5678
   ↓
Docker Compose
   ├── n8n
   └── PostgreSQL
```

### Why NGINX runs on the VM (not in Docker)

* Stable WebSocket handling
* Clean TLS termination
* Predictable reverse proxy behavior
* Easier compliance validation

# Security Model

### Secrets

* PostgreSQL password → generated by Terraform
* n8n encryption key → generated by Terraform
* Stored in Azure Key Vault
* Never stored in Terraform state in plaintext
* Never stored in Git
* Never hardcoded in Docker

### Secret Access

The VM uses **Managed Identity** to authenticate to Azure.

At boot:

1. Azure CLI authenticates via `az login --identity`
2. Secrets are pulled from Key Vault
3. Environment variables are exported
4. Docker Compose starts with those secrets

No credentials exist on disk.

# Boot Process (What Actually Happens)

After `terraform apply`:

1. VM is created
2. Managed Identity is attached
3. Key Vault is created
4. Secrets are generated and stored
5. Custom Script Extension runs `install.sh`
6. `install.sh`:

   * Installs Docker
   * Installs Docker Compose
   * Installs NGINX
   * Installs Azure CLI
   * Configures self-signed SSL
   * Creates systemd service (`n8n.service`)
7. `systemd` starts `start-n8n.sh`
8. `start-n8n.sh`:

   * Logs into Azure using Managed Identity
   * Fetches secrets
   * Exports environment variables
   * Runs `docker-compose up` (attached mode)
   * Containers start
9. NGINX reverse proxies HTTPS → n8n

Result:

You can immediately open:

```
https://<VM_PUBLIC_IP>
```

# Important Implementation Details

### Azure CLI on the VM

The VM installs Azure CLI during provisioning because the runtime script depends on:

```
/usr/bin/az login --identity
```

Without Azure CLI installed, secret retrieval would fail.

---

### systemd Service

A systemd service (`/etc/systemd/system/n8n.service`) ensures:

* n8n starts on boot
* Docker is required before startup
* If containers crash, systemd restarts the service
* Logs are written to:

  * `/var/log/n8n-start.log`
  * `journalctl -u n8n`

This prevents restart loops and 502 errors caused by detached Docker processes.

---

### Docker Compose Behavior

The service runs:

```
docker-compose up
```

(not `-d`)

This is intentional.

Why?

* systemd must own the foreground process
* Avoids container restart loops
* Ensures clean service lifecycle management

# Deployment

## Prerequisites

* Azure subscription
* Azure CLI logged in (`az login`)
* Terraform ≥ 1.5
* SSH keypair

Generate one if needed:

```bash
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -C "n8n-azure"
```

---

## Configure terraform.tfvars

```hcl
resource_prefix     = "n8n"
location            = "eastus"
vm_size             = "Standard_B2s"
admin_username      = "azureuser"
ssh_public_key_path = "~/.ssh/id_ed25519.pub"
```

---

## Deploy

```bash
terraform init
terraform apply
```

Wait for completion.

No SSH required.

# Common Issues (Resolved During Build)

### 502 Bad Gateway

Cause:

* n8n not binding correctly
* Docker Compose running detached under systemd
* Environment variables lost due to sudo context

Fix:

* Run docker-compose in attached mode
* Let systemd own process
* Remove unnecessary sudo inside service script
* Ensure ports expose 5678:5678

---

### Azure CLI Not Found

Cause:

* CLI not installed on VM

Fix:

* Install Azure CLI inside `install.sh`
* Verify with `command -v az`

---

### Docker Permission Errors

Cause:

* Non-root user not in docker group

Fix:

* systemd runs service as root
* Docker accessed without interactive sudo
